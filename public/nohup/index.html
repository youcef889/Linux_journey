The nohup (“no hang-up”) command is a simple yet powerful tool in Unix/Linux that allows you to run a command immune to SIGHUP (hangup) signals. In practice, this means that if you start a long‐running process and then log out (or your terminal closes), the process will continue running in the background. Below is a detailed overview of what nohup is, when and why you’d use it, and how to use it effectively.

1. What Is nohup and Why Use It?
#

  • Purpose: By default, most shells send a SIGHUP (signal hangup) to all their child processes when the controlling terminal closes (e.g., you type exit, close an SSH session, or press Ctrl+D). Unless the process deliberately traps or ignores SIGHUP, it will terminate.

  • nohup’s Role: When you prefix a command with nohup, it arranges for that command to ignore SIGHUP. Thus, even if you log out or your terminal dies, the process keeps running.

  • Typical Use Cases:

    • Long‐running jobs (e.g., data processing, model training, backups, server scripts) that must survive after you disconnect.
    • Remote servers: Start a process over SSH, then safely close the SSH session.
    • Prevent accidental termination: You might accidentally close the terminal; nohup ensures the process keeps going.

2. Basic Syntax
# 

nohup COMMAND [ARGUMENTS…] [ <&/dev/null ] [ >& OUTFILE ] &

  • nohup: the command itself.

  • COMMAND [ARGUMENTS…]: whatever program or script you want to run.

  • &: typically appended at the end to push the process into the background immediately.

  • Output redirection:

    • By default, if you don’t explicitly redirect standard output (stdout) (and standard error), nohup writes both stdout and stderr to a file named nohup.out in the current directory (or to ~/nohup.out if it cannot write to the current directory).
    • You can override that by explicitly redirecting output (→ see “Output Handling” below).

  • Input redirection:

    • If your command tries to read from standard input, it will see EOF immediately (unless you explicitly tie it to something else). Commonly, you’ll redirect stdin from /dev/null to avoid it hanging for input:

      nohup myscript.sh </dev/null &

In short:

nohup my_command arg1 arg2 … > my_command.log 2>&1 &
  • > my_command.log redirects stdout into my_command.log.
  • 2>&1 redirects stderr to stdout (so all output ends up in the same logfile).
  • & sends the entire thing to the background.

3. How nohup Actually Works
#

  1. Ignores SIGHUP Internally, nohup calls signal(SIGUP, SIG_IGN), telling the OS to ignore hangup signals for the child process.

  2. Redirects Output If your terminal (pty) disappears, any unreadable/writable file descriptor can cause a process to be killed or blocked. nohup ensures that:

    • If stdout isn’t redirected, it opens (or creates) nohup.out in the current directory (or ~/nohup.out as fallback) and makes that the new stdout.
    • Likewise, stderr is redirected to the same place as stdout (unless you override it).

  3. Runs the Command After handling signals and redirections, nohup execs your command with its altered signal handling and file descriptors.

4. Common Usage Patterns and Examples
#

Below are some practical examples that illustrate how to use nohup in everyday scenarios.

4.1. Running a Script in the Background
#

Suppose you have a Python data‐processing script long_job.py that may take several hours. You want it to keep running even if you close your SSH session:

nohup python3 long_job.py > long_job.log 2>&1 &

  • Explanation:

    • nohup python3 long_job.py: Launches python3 long_job.py immune to hangups.
    • > long_job.log: Redirects stdout into long_job.log.
    • 2>&1: Redirects stderr to the same file (long_job.log).
    • &: Puts the process in the background immediately, giving you back your shell prompt.

  • Result:

    • The script’s both stdout/stderr get written into long_job.log.
    • After you press Enter, you see a job ID and PID printed, but then you can safely log out of SSH.
    • Even if your terminal closes, the Python script keeps running.

4.2. Default Output File (nohup.out)
#

If you omit explicit redirection, nohup writes to nohup.out:

nohup ./backup_database.sh &

  • Effect:

    • A file named nohup.out is created in your current directory.
    • Both stdout and stderr from backup_database.sh will be appended to nohup.out.
    • If nohup.out cannot be created in your CWD (permissions), it tries ~/nohup.out.

4.3. Redirecting Only stderr or Only stdout
#

If you want to separate standard output and error:

nohup my_command > output_only.log 2> errors_only.log &
  • > output_only.log → stdout goes to output_only.log.
  • 2> errors_only.log → stderr goes to errors_only.log.

4.4. Redirecting stdin from /dev/null
#

Some scripts may occasionally try to read from stdin. To avoid accidental blocking, you can tie stdin to /dev/null:

nohup my_interactive‐ish_command </dev/null > out.log 2>&1 &
  • < /dev/null: Immediately makes standard input be “end‐of‐file.” The process won’t pause waiting for you to type.

5. Checking and Managing a nohup Process
#

Once you’ve launched a job with nohup … &, you’ll want to:

  1. Find the PID / Job number

    • When you press Enter after starting with &, the shell usually prints something like:

      [1]  12345

      where 1 is the job ID (in this shell session) and 12345 is the PID.

    • If you missed that, use ps, pgrep, or ps –f u $USER | grep my_command to locate it.

  2. See If It’s Still Running

    ps -p 12345   # or: ps aux | grep my_command

  3. Attach to Logs

    tail -f long_job.log

    or

    tail -f nohup.out

    depending on where you redirected output.

  4. Killing the Job If you need to stop it:

    kill 12345         # sends SIGTERM
kill -9 12345      # sends SIGKILL (forceful)

    (Replace 12345 with the actual PID.)

  5. Disowning Jobs (Alternative Approach) If you start a process without nohup, you can use your shell’s built‐in disown to remove it from the shell’s job table so it doesn’t receive SIGHUP:

    long_running_command &
disown -h %1    # tells shell not to send SIGHUP to job #1

    However, unlike nohup, you still need to manually redirect stdout/stderr or else the process may hang or die if it tries to write to a closed terminal.

6. When Not to Use nohup
#

  • Interactive Processes: If your process requires interactive user input any time in its execution, nohup might not be appropriate. Redirecting stdin to /dev/null means it’ll never get interactive input.

  • Long‐term Supervision: For extremely long‐lived services (e.g., daemons, web servers), it’s often better to use a proper init system (systemd service, Upstart, SysV init script) or a process‐supervision tool (Supervisor, tmux/screen, forever, pm2 for Node.js, etc.).

    • systemd (modern Linux distros) or supervisord can auto‐restart your process if it crashes.
    • nohup alone won’t restart a process if it dies— it merely ensures it isn’t killed on logout.

7. Combining nohup with Other Tools
#

  • screen or tmux: If you want to re-attach later to the running process’s terminal (e.g., watch real‐time output, send input), you might prefer starting it inside a screen or tmux session instead of or in addition to nohup.

    screen -S mysession
# inside screen:
./my_server.sh
# Press Ctrl+A, D to detach
# even if you log out, screen keeps it alive

    You can later re‐attach with screen -r mysession.

  • setsid: Another way to decouple a process from the controlling terminal is to use setsid:

    setsid my_command > out.log 2>&1 &

    This creates a new session and detaches from the terminal—but unlike nohup, you must manually arrange to ignore SIGHUP or redirect properly.

  • & + disown (in bash, zsh, etc.):

    my_command &
disown

    This disowns all jobs in the current shell so they won’t receive SIGHUP. But you still need explicit output redirection (otherwise writing to a closed TTY may cause the process to be killed).

8. Quick Reference / Cheat Sheet
#

## 9. Practical Example: Launching a Flask Development Server

Imagine you’re working on a Flask app called `app.py` and want to let it run on port 8000 even after you disconnect:

```bash
# 1. Navigate into your Flask project directory
cd ~/projects/myflaskapp

# 2. Launch Flask via nohup:
nohup flask run --host=0.0.0.0 --port=8000 > flask-server.log 2>&1 &

# Shell prints, e.g.: [1] 23456
# - PID 23456 is your Flask server.

  • View logs while it’s running:

    tail -f flask-server.log

  • Stop the server later:

    kill 23456

10. When You Don’t See Output in the Expected Place
#

  • If you run nohup my_command without explicitly redirecting stdout/stderr, but after some time you don’t see a nohup.out in your current directory:

    1. Check permissions: Maybe you don’t have write permission in that directory, so nohup wrote to ~/nohup.out instead.

    2. Check your working directory: Are you sure you ran it from where you think? nohup resolves nohup.out relative to CWD at the moment of launch.

    3. Verify the process started:

      ps aux | grep my_command

      If it’s not there, maybe it failed immediately (e.g., syntax error). If it is there, note its PID and check ls -l nohup.out ~/nohup.out.

  • Always explicitly redirect if you care exactly where logs go:

    nohup my_command > /var/log/my_command.log 2>&1 &

11. Summary
#

  • nohup prevents a child process from receiving the SIGHUP signal when its controlling terminal closes.

  • By default, it redirects output to nohup.out (or ~/nohup.out), but you can and often should explicitly redirect stdout/stderr to a file of your choosing.

  • Most often used with & to push a process to the background immediately, e.g.:

    nohup some_long_job > some_long_job.log 2>&1 &

  • For truly “bulletproof” long‐running daemons or for processes you want to re‐attach to interactively later, consider using a terminal multiplexer (screen, tmux) or a system/service manager (e.g., systemd, supervisord).

  • Remember to check on your nohup processes via ps, tail your log files, and kill them when you’re done.

With this knowledge, you should be able to launch virtually any long‐running or “daemon‐style” job safely, without worrying that closing your laptop or SSH session will terminate it.
